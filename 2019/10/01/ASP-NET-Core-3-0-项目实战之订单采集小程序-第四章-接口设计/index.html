ASP.NET Core 3.0 项目实战之订单采集小程序 第四章 接口设计

发表于 2019-10-01 更新于 2019-10-23 分类于 .NET Core 阅读次数：
本文字数： 2.2k 阅读时长 ≈ 3 分钟

Web API 接口是前端和后端通讯的主要方式，所以在做前后端分离的项目时，需要先设计好 API 接口。

尤其是当前端和后端的开发人员不是同一个人时，那么在编码前，就必须约定好接口规范。否则，你做你的，我做我的，最后发现对接不上，那就很糟糕了。

即便这篇教程里的案例很简单，我也仍然建议大家先把接口设计好，这样就不会在编码时找不到方向，随性而为，甚至写着写着，接口就偏离了 RESTFul 风格。

用户登录

简要描述

请求方式

  • POST

请求参数

参数名 必须 类型 说明
code 是 string 钉钉免登授权码

订单相关

历史订单查询

简要描述

  • 查询用户历史订单，需要鉴权，默认查询前20条

请求 URL

  • /api/orders

请求方式

  • GET

请求参数

参数名 必须 类型 说明
pageIndex 否 int 页码，默认查询第 1 页
pageSize 否 int 页尺寸，默认查询 20 条
state 否 int 订单状态，0、未确认，1、已确认，默认查询全部

返回正确示例

订单修改

简要描述

  • 修改指定编号订单内容，需要鉴权，只能修改状态为「未确认」订单的「日期」字段

请求 URL

请求方式

请求参数

参数名 必须 类型 说明
id 是 int 要修改的订单编号
date 否 string 订单日期，格式：“yyyy-MM-dd”，例：“2019-09-30”

409 Conflict 资源发生冲突
